## Problem Statement

製造業の設計者(製品・金型・設備)は、過去のトラブル事例(過去トラ)やベテランの設計知見が
Excel・PPT・PDF・人の頭の中、そして **CADデータの改訂履歴(△注記)の中**に散在しているため、
設計の最中にそれらを参照できず、**既知の失敗を再発させている**。

具体例(デモのシードデータそのもの): 2015年に「板厚8mmでは強度不足」と△1で学んだ教訓が
共有されず、2020年モデルで別の設計者が軽量化のため板厚を8mmに戻し、**全く同じ強度不足を再発**。
手戻りと軽量化断念のコストが発生した。2025年モデルでは2015年と同じ「Rダラし」の失敗が
再発しかけている。

これはチーム(トヨタ所属)が社内で実際に観測している痛みであり、仮説ではない。
ベテラン退職により知見の担い手が物理的に消えていく構造的な時限性もある。

## What Makes This Cool

**「後ろで見てくれている先輩」**(ユーザー本人の言葉)。

設計者が過去知見的にまずい設計操作をした**瞬間**に、AIエージェントが割り込んで警告する:
- どの過去の△(改訂)に該当するか
- 出典はどの文書か(報告書・PPT)
- 誰がその問題を解決したか
- 過去に再発した時に何が起きたか

質問→回答のチャットボットではない。検索でもない。YC RFS「Company Brain」が明言する
「**会社の動き方の生きた地図**」と「**executable skills files**」の文字通りの実装。

さらに終盤のフライホイール演出: 設計者が変更を確定すると、AIが
「この変更の理由は強度不足への対応ですよね？」とワンタップ確認質問し、
「はい」と答えるだけで**知識グラフに新ノードがその場で増える** —「脳が今、育った」。

## Premises

リポジトリの [PREMISES.md](PREMISES.md) で合意済み。要約:

1. **問題の実在性**: 知見は文書だけでなくCADの△改訂履歴の中にも残っている。トヨタで観測済みの実在の痛み
2. **デモの核**: 合成データ + Web 3Dビューア疑似CAD。「設計操作 → 先輩AIの根拠付き警告」の瞬間が主役
3. **差別化と3段階ビジョン**(第二意見を受けて修正済み):
   - 今日 = 読み取りパス(△ → 構造化 → 警告)
   - 次 = △をリンクキーにして報告書・PPTを知識本体としてコンパイル
   - 将来 = 書き込みパス(設計変更の瞬間にAIが確認質問 → 答えるだけで次車種のための構造化データが貯まる = living map / データ・フライホイール)
4. **今日のスコープ**: 知識構造化パイプライン + 3Dビューア警告UI + 90秒英語デモ + デプロイ。実CAD統合・本番データ接続はやらない
5. **市場観点**: 「製造業 × 設計知見の再発防止」という狭いくさびから語る

## Cross-Model Perspective

第二意見(独立Claudeサブエージェント / Codexはバイナリ破損でフォールバック)の要点:

1. **Steelman**: 「死んだ組織知から設計時のライブ強制へのコンパイラ。△注記は『何が壊れ、なぜ壊れたか』の最も密度が高い記録 — 直した本人が、壊れた形状そのものに書き残したもの — であり、誰もクエリできたことがない。汎用RAG製品には到達不能な、ドメインアクセスで守られた本物のCompany Brainのくさび」
2. **最重要指摘**: 製品の本体は抽出パイプラインではなく「割り込みの瞬間」。**間違って割り込む先輩は永遠に無視される — 偽陽性率がこの製品の生死を決める**
3. **前提3への挑戦**: △注記単体は情報量が薄い。実図面OCRはCADDiの領域で少人数の主戦場にすべきでない → △はリンクキー、報告書/PPTが知識本体(ユーザーが受容し、フライホイール構想へ昇華)
4. **実装提言(採用)**: デモの照合はLLMでなく**決定的ルールマッチ**(幻覚ゼロ・遅延ゼロ)。LLMは①照合済み知見の自然言語説明と②ライブ抽出演出のみに使う。知識ベースは手書き構造化レッスン約30件のJSONで十分。ベクタDB・RAGフレームワーク不要

## Approaches Considered

詳細は [APPROACHES.md](APPROACHES.md)。

### Approach A: 全静的ワンページ(最小構成)
静的HTML + three.js のみ、実行時LLMなし。Effort S / Risk Low / Completeness 5/10。
確実に動くが「AIプロダクト」の主張が弱い。

### Approach B: フルスタック + LLM演出(理想形)
Vite/Next.js on Vercel。ビューア + 決定的マッチ警告 + Claude APIによる説明生成 +
「コンパイラ」ライブ抽出シーン + 知識グラフ。Effort M / Risk Medium / Completeness 7/10。

### Approach C: フライホイール実演(創造的・終盤ビート)
A/Bに書き込みパスを追加: 変更確定 → AIの確認質問 → ワンタップ回答 → グラフに新ノード。
Effort +S / Risk Low-Med。

## Recommended Approach

**B + C(フル機能帯)— ユーザー承認済み。**

理由: ゴールは「優勝 + 起業視野」。優勝にはBのライブ抽出と知識グラフの絵、
投資可能性にはCのフライホイール(データ・モートの実演)が直撃する。

**実装原則**:
- 警告の照合は決定的ルールマッチ。警告カードの文言は事前生成の静的テキスト。
  LLMの実行時利用はライブ抽出シーンとフライホイール確認質問の2つのみ
- 偽陽性をデモで一度も見せない(的外れ警告は製品への信頼を殺す)
- デモ状態はURLパラメータでシードし、90秒フローを決定的にする
- ネットワーク依存シーン(ライブ抽出)には事前録画のバックアップを用意

**退避ラダー**(時間が押したら段階的に降格):
B+C フル → B(抽出シーンを事前録画に) → A+C → A。
締め切り90分前に動いているものが「今日の製品」。以降は磨きとリハのみ。
**デプロイのフォールバック**: Vercel 不可・回線断の場合はローカルサーブ(`vite preview`)+
スマホテザリングで見せる。デモPCには常にローカルで動く最新ビルドを置いておく。

**技術スタック(推奨)**:
- フロント: Vite + three.js。**3Dモデルはプリミティブのプロシージャル合成で作る
  (Cylinder + Box 等。STL/GLB調達はしない — 読み込んだSTLは板厚やRを動的に変えられないため)**。
  R(フィレット)の視覚表現はトーラス/円柱セグメントの差し替えによる**擬似表現で割り切る**
  (見た目が変わればよい。CSG/カスタムジオメトリは踏まない)。
  操作UIは**板厚とRの2つだけ**(穴位置は発火ルールにもビート表にも登場しないためカット。
  バッファ時間が余った場合のみ追加検討)
- 知識ベース: 手書き構造化レッスン **10〜12件**(シードの9△ + 汎用ダミー数件)の単一JSON。
  スキーマ: `category / 部位 / 操作 / パラメータ / 閾値 / outcome / 出典文書 / 出典△ / 担当者 / 年`
  (デモで発火するのは下記2ルールのみ。件数を盛る必要はない)
- 照合: クライアント側の決定的ルールエンジン。**先輩カードは実行時LLMを使わない** —
  R1/R2の先輩口調の文言は**事前生成して静的に埋め込む**(内容は事前に完全確定しているため)。
  これにより警告シーンは完全にネットワーク非依存になり、「2秒以内」基準を無条件に満たす
- LLM: Claude API — **Vercel Serverless Functions 経由でプロキシ(APIキーは環境変数。
  クライアント直呼びはキー露出のため禁止)**。用途は①ライブ抽出シーン
  ②フライホイール確認質問の**2つのみ**。タイムアウト時は静的文言にフォールバック。
  **APIキーが無い場合のライブ抽出の代替は録画ではなく、スクリプト化された
  タイプライター風の疑似アニメーション**(入力テキスト→JSON整形の演出。録画対象がそもそも存在しないため)
- 知識グラフ: **vis-network を npm 依存として Vite にバンドル**(CDN読み込みは回線断時の
  ローカルビルドを壊すため禁止)。ノードは固定座標の手置き(~15ノード)。
  インタラクティブ探索は作らない — 固定レイアウト + フライホイール時の「ノード追加演出」のみ
- デプロイ: Vercel(審査員が触れるライブURL + スライドにQRコード)

**デモで発火する警告ルール(この2つが全て。これ以外は発火させない = 偽陽性ゼロ)**:

| # | 発火条件 | 照合先 | 先輩カードの表示内容 |
|---|---|---|---|
| R1 | 板厚スライダーが **10mm未満** になった瞬間 | 2015 △1 / 2020 △2(再発) | 「板厚10mm未満は締結部の応力集中で2度失敗しています」+ 2015 △1(山田)・2020 △2(鈴木、軽量化断念)+ 出典報告書 |
| R2 | R値を **R5→R3 に変更**した瞬間 | 2015 △2→△3 | 「Rダラしは剛性低下で手戻りになりました。逃がし形状(2020 △3で成功)を推奨」+ 当時の担当・佐藤 |

**90秒デモのビート表(実装より先に確定。実装はこのスクリプト駆動で行う)**:

| 秒 | ビート | 画面 |
|---|---|---|
| 0-15 | 問題提起(英語): 「2015年の教訓が2020年に再発した。知見は図面の△の中で眠っている」 | flange-revision-history の再発チェーン図 |
| 15-40 | **警告の瞬間**: 2025年モデルでR5→R3操作 → 先輩カードが割り込む(R2発火) | 3Dビューア + 先輩カード |
| 40-60 | コンパイラ: 報告書テキスト投入 → 構造化レッスンJSONが生成される(ライブ / キー無し時は疑似アニメ) | 抽出シーン |
| 60-80 | フライホイール: 先輩カード上の**「逃がし形状を適用」ワンクリックボタン**を押す(警告に従う筋書き。ジオメトリはプリミティブ差し替えの擬似表現) → 「理由は干渉回避ですよね？」→ はい → グラフにノード追加 | 知識グラフ |
| 80-90 | 締め(英語): 「設計するほど、脳が育つ。Every factory needs this brain.」+ QR | ライブURL/QR |

## Business Overview(審査提出「ビジネス概要」の骨子)

- **立場**: スタートアップを立ち上げる創業メンバー。トヨタで働いているからこそ
  課題の解像度が高く、最初の顧客への人脈もある — これがこのチーム固有の不公平な優位性
- **最初の顧客**: **トヨタ(またはトヨタ関連サプライヤー)の設計部門**(製品/金型/設備設計者)。
  中の人間としての課題観測と人脈から、PoC の入口を最短で確保できる
- **どう売るか**: 部門単位のSaaS(設計者数×月額)。PoCは「過去トラ再発1件の防止コスト
  (手戻り工数+量産遅延)」をROIの物差しにする — 再発1件で年間ライセンス費を回収できる価格設定
- **横展開**: 設計知見(△)→ 工程知見 → 品質保証へと同一構造(散在知識→実行可能チェック)を拡張。
  自動車に限らず製造業全体に共通する問題であり、書き込みパス(フライホイール)により
  顧客のデータ資産が増え続け、スイッチングコストがモートになる
- **最終ビジョン**: **世界中の製造業の Company Brain を弊社が達成する**。
  トヨタ → 自動車サプライチェーン → 製造業全体、の順で狭いくさびから広げる

## What I noticed about how you think

- 「質問して回答するパターンじゃ面白くないし価値がない」と最初の提案を切り捨て、
  「**後ろで見てくれている先輩**」という一文で製品の本体を言い当てた。
  プロダクトの感情的な芯を自分の言葉で定義できるのは作り手として稀な資質
- 第二意見が前提3を攻撃した時、防御でも全面降伏でもなく
  「**指摘の通り△自体には情報量が少ない。将来的にはリンクキーだ**」と受け止めた上で、
  「答えるだけで次車種のための構造化データが貯まる」という
  誰も提示していなかったフライホイール構想を自分から出した。挑戦を受けて設計が強くなる型
- セッション前から flange-revision-history.md を用意していた。世代横断の再発パターン、
  ハイライトシーン、スキーマ示唆まで書き込まれた一級のシードデータ。構想でなく手が動いている
- 「製造業向け、CADを扱う、ただし最初はCADにフォーカスしたい」— 対象の絞り込みを
  自分で宣言できている。広げたがるのが普通のハッカソンで、狭められるのは強い
