# PREMISES — オフィスアワーで確認する5つの前提条件

このファイルは /office-hours セッションの前提確認用。設計ドキュメントはこの前提の上に作られる。

## 前提1: 問題の実在性 【修正済み】

製造業の設計者(製品・金型・設備の設計者)は、過去のトラブル事例(過去トラ)やベテランの設計知見が **Excel・PPT・PDF・人の頭の中に散在している**ため、設計の最中にそれらを参照できず、**既知の失敗を再発させている**。

さらに重要なこととして、**知見は文書だけでなくCADデータそのものの中にも残っている**。
図面の改訂履歴(△)には「△1 強度確保のため板厚追加」のような変更理由が刻まれており、
これは世代をまたいで蓄積された設計判断の記録である
(実例: [design/mocks/flange-revision-history.md](../design/mocks/flange-revision-history.md) —
2015年の「板厚8mmでは強度不足」という△知見が共有されず、2020年に全く同じ失敗が再発)。

これはトヨタの中で実際に観測している痛みである(仮説ではない)。

→ 審査で「本当にある問題なの？」と聞かれた時の根拠になる。

## 前提2: デモの核

デモは「**リアルな合成データ + Web 3Dビューアの疑似CAD**」で作る。

主役の瞬間は、**設計操作をした瞬間に先輩AIが根拠付きで警告する**シーン:

- どの過去トラ事例に該当するか
- どの文書(Excel/PPT/PDF)が根拠か
- 誰がその問題を解決したか

まで提示する。「後ろで見てくれている先輩」の体験。

## 前提3: 差別化の主張と最初の機能 【第二意見を受けて再修正済み】

**製品ビジョン(3段階)**:

1. **今日作る(読み取りパス)**: CADデータ内の△改訂履歴を構造化データに変換し、
   設計操作の瞬間にAIエージェントが過去知見を根拠に警告・提案する
2. **次(統合)**: △はリンクキーとして扱い、△が指す先のトラブル報告書・設計審査PPTを
   知識の本体としてコンパイルする(△単体は情報量が少ないという第二意見の指摘を受容)
3. **将来(書き込みパス = フライホイール)**: 設計者が新プロジェクトで変更を加えた瞬間に、
   AIが「理由は強度不足ですよね？」と軽い確認質問をし、**答えるだけで
   次車種のための構造化データが自動的に貯まる**仕組み。
   知識は使うほど・設計するほど増える = RFS の言う「living map(生きた地図)」

**差別化ロジック**:
- 既存のDFMチェッカー(CoLab AutoReview 等)は「形式化済みの社内標準」へのチェック。
  うちは「△改訂履歴・散在文書という**形式化されていない知見**から警告を自動コンパイル」
- さらに既存ツールは全て**読むだけ**。うちは設計行為そのものを知識化する
  **書き込みループ**を持つ(= データ・フライホイール = 投資家向けのモート)
- 本体は「知見 → 実行可能な設計チェック」への**コンパイラ**。
  YC RFS の "executable skills files" の文言そのもの

**デモ実装の注意(第二意見より採用)**:
- デモの照合はLLMではなく**決定的ルールマッチ**(幻覚ゼロ・遅延ゼロ・確実に動く)。
  LLMは照合済み知見の上の自然言語説明の生成のみに使う
- 「間違って割り込む先輩は永遠に無視される」— 偽陽性率がこの製品の生死を決める。
  デモでも的外れ警告は一度も見せてはいけない

## 前提4: 今日のスコープ 【修正済み】

今日(5時間)で作るのは以下の4つ:

1. 知識構造化パイプライン(△改訂履歴・散在文書 → 構造化知識)
2. 3Dビューア + 警告UI
3. 90秒英語デモ
4. **デプロイ**(審査員が自分のブラウザで触れるURLを用意する)

**やらないこと**(設計ドキュメントに将来課題として明記):

- 実CAD(CATIA/NX等)への統合
- 本番データ接続

## 前提5: 市場観点の語り方

審査提出物では「すべての会社が Company Brain を必要とする」という一般論**ではなく**、
「**製造業 × 設計知見の再発防止**」という具体的なくさび(狭い入口)から語り、
そこから横展開する構成にする。

## 参考: ランドスケープ調査の結果

- **CoLab AutoReview** — AIが3Dモデルにピンを打ち「違反した社内標準」を引用して指摘。最も近い競合
- **Leo AI** — PDM/PLM/CAD/PDF を横断する「知識レイヤー」
- **CADDi** — 散在図面を検索可能な資産に(調達・類似図面検索が中心)
- **DraftAid (YC)** — 3Dモデル→図面のAI自動生成

**EUREKA(通説の反転)**: 既存ツールが形式化済み知識しか扱わないのは、「非構造文書→実行可能ルール」の変換が技術的に不可能だったから。LLMで初めて可能になった。ここがこのプロダクトの本体。
